




* 核心概念：MongoDB 中的分片  
  
分片是 MongoDB 實現水平擴展的一種方式——將大型資料集拆分到多個伺服器（分片）上，從而避免單一機器承擔全部負載。分片鍵是用來決定文件如何在分片間分佈的一個或多個欄位。 mongos路由器位於應用程式和分片之間，負責將查詢定向到正確的位置。  


Ranged Sharding－根據實際分片鍵值範圍散佈文件。相鄰值保留在相同或相近的分片中，從而保持自然順序。  
```sql
sh.shardCollection ( "myDatabase.orders " , { order_date : 1 } )   
```

Hashed Sharding－在分發資料之前對分片鍵套用雜湊函數，有意分散文件以實現均勻分佈。  

```sql
sh.shardCollection ( "myDatabase.orders" , { order_date : "hashed " } )   
```










負載平衡器是基於資料區塊數量而非大小或查詢負載來工作的，並且控制負載平衡視窗和閾值是管理叢集效能的關鍵。  





[題目]  
In MongoDB sharding, the balancer is responsible for migrating chunks across shards to ensure even data distribution. In the context of chunk management and balancing, what is the PRIMARY condition that triggers the balancer to begin migrating chunks from one shard to another? (difficulty: hard)

[選項]  
A. The mongos router detects uneven query load distribution across shards and signals the balancer to redistribute chunks.
B. The difference in the number of chunks between the shard with the most chunks and the shard with the fewest chunks exceeds the migration threshold.
C. The total number of documents in the collection exceeds a predefined global threshold.
D. A single chunk grows beyond the configured chunk size limit, causing an automatic split and immediate rebalancing.
	
[正確答案]：
B	
	
[核心考點]： 	
	
[詳細解析]：
本題考查 MongoDB 分片平衡器的觸發機制。平衡器是基於各分片間「Chunk 數量的差異」是否超過特定的「遷移閾值（Migration Threshold）」來決定是否啟動數據遷移，而非依據查詢負載或文件總數。

A. 錯誤原因：`mongos` 路由主要負責分發查詢，但不會監控查詢負載來通知平衡器進行數據遷移，平衡器只關注資料在分片間的分佈狀態（Chunk 數量或大小差異）。

B. 正確原因：MongoDB 平衡器會定期檢查分片間的 Chunk 分佈，當含有最多 Chunk 的分片與含有最少 Chunk 的分片之間的數量差超過遷移閾值時，就會觸發遷移。

C. 錯誤原因：平衡器的觸發與集合內的文件總數無關。即使集合擁有數十億個文件，只要各分片間的資料分佈是均勻的（未達閾值），就不會觸發平衡器。

D. 錯誤原因：當單個 Chunk 超過大小限制時，會觸發 Chunk 分裂（Split），這是在同一分片上的元數據操作，並不會「立即」自動觸發跨分片的數據遷移。
