rs.status()：STARTUP→ STARTUP2（初始同步）→→ RECOVERING（SECONDARY健康）。  

ROLLBACK（回滾）狀態：這發生在「舊 Primary 突然掛掉，但它本地有幾筆寫入還沒同步給其他人；當它重新上線變成 Secondary 時，必須把這幾筆未同步的資料剔除」，這與 Oplog 被覆蓋無關。  
RECOVERING（恢復中）狀態：這是一個暫時性的狀態（例如節點剛啟動正在檢查資料、或正在建立索引），不具備代表「進度過期」的特殊含意。  



# All Read Concern Levels Compared

在 MongoDB 中，讀取關注級別（Read Concern）決定了你所讀取的資料「新鮮度」與「持久度」保證。

| Read Concern | 核心特性描述 | 適用場景與效能表現 | 交易（Transaction）支援 |
| :--- | :--- | :--- | :--- |
| `local` | 1. 僅返回被查詢節點上的當前資料，不保證該資料已同步（複製）到其他節點。 <br>2. 若發生 Primary 節點故障移轉（Failover），此資料**有可能會被回滾（Rollback）**。 <br>3. 單機（Standalone）與副本集（Replica Set）在非交易讀取時的**預設值**。 | * **最快的讀取速度。** | 不適用於多文件交易。 |
| `available` | 1. 行為與 `local` 類似，但專為**分片群集（Sharded Clusters** 進行優化。<br>2. 在分片環境中，**有可能會讀取到孤立文件（Orphaned Documents）**。 | * **延遲最低**，但提供最少的一致性保證。 | 不適用於多文件交易。 |
| `majority` | 1. 僅返回已被大多數（Majority）副本集成員確認且已寫入預寫日誌（Journal)的資料。<br>2. 具備 **防回滾（Rollback-safe）** 安全特性。 | * **強韌性與效能的最佳平衡點**。<br>* 適合關鍵業務中，對資料一致性與持久性要求極高的應用。 | 支援。 |
| `linearizable` | 1. 提供**最強的一致性（Strongest Consistency）**。<br>2. 確保讀取前，所有多數確認（Majority-acknowledged）的寫入都已反映。 | * **僅限於單一文件讀取（Single-document reads）**。<br>* **延遲極高**，不適合作為一般用途的讀取設定。 | 不支援。 |
| `snapshot` | 1. 提供交易開始那一刻的**一致性快照（Consistent Snapshot）**，其基礎來自於多數已提交（Majority-committed）的數據。 | * **專為多文件交易（Multi-document Transactions）設計**。 | **必須**在交易中使用。 |

### 💡 備考速記口訣

> * 追求極速與分片相容 $\rightarrow$ 選 `local` / `available`
> * 防止資料回滾（安全至上） $\rightarrow$ 選 `majority`
> * 強迫即時一致（單筆讀取） $\rightarrow$ 選 `linearizable`
> * 執行 ACID 多表交易 $\rightarrow$ 唯一指名 `snapshot`


當一個主節點無法進入時，剩餘的投票成員會進行選舉，選出一個新的主節點。選舉演算法主要考慮兩個因素：  
  
1. 每個成員配置的優先順序   
2. 每個成員的 oplog 相對於副本集最後已知狀態的時效性  

* 基本情境－初選失敗與選舉:  
1. 想像一個 3 節點副本集：節點 A（主節點）、節點 B（從節點）、節點 C（從節點）。  
2. 節點 A 崩潰或無法存取。  
3. 節點 B 和節點 C 大約 10 秒後停止接收來自節點 A 的心跳（預設 electionTimeoutMillis）。  
4. 節點 B 和節點 C 進行選舉。兩者都會檢查其操作日誌的有效性和配置的優先順序。  
5. 假設節點 B 的優先權為 2，節點 C 的優先權為 1，且兩者的操作日誌都是最新的。節點 B 獲勝，因為它具有更高的優先權。  
6. 節點 B 成為新的主節點。節點 C 仍然是從節點，現在從節點 B 複製資料。  
7. 所有寫入操作均透過節點 B 恢復。  
8. 當節點 A 恢復時，它會作為輔助節點重新加入，並透過 oplog 複製追趕進度。  


* 高級場景 — 操作日誌貨幣覆蓋優先權
  
考慮一個 3 節點副本集：節點 A（主節點，優先權 1），節點 B（從節點，優先權 2），節點 C（從節點，優先權 1）。  
網路分區導致節點 A 短暫隔離。在此期間，節點 A 執行了 50 次新的寫入操作，節點 B 尚未複製這些操作。節點 C 已複製了分區之前的所有操作。  

1. 選舉已觸發。節點 B 的優先權較高（2），但其操作日誌落後於節點 C 的操作日誌。  
2. MongoDB 的選舉演算法會阻止節點 B 獲勝，如果另一個符合條件的節點（節點 C）擁有節點 B 尚未複製的更新的 oplog。  
3. 節點 C 被選為新的主節點，因為它的 oplog 是最新的——即使它的優先權較低。  
4. 這樣可以防止資料遺失：如果節點 B 被選為節點並開始接受新的寫入操作，那麼它將遺失這 50 個操作。
