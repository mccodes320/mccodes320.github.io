[題目]   
What is considered a best practice when naming fields in MongoDB documents? (Select one.)      
      
[選項]   
A. Using short, abbreviated names to conserve space on disk.   
B. Re-using field names within a document to optimize indexes.   
C. Using descriptive, unique names.   
D. Using generic names.   

[正確答案]：   
C   
   
[核心考點]：   
   
[詳細解析]：   
本題考查 MongoDB 文件欄位命名的最佳實踐。MongoDB 的文件結構即為其資料型態（Schema），清晰的欄位命名能大幅提高程式碼的可讀性與可維護性。雖然 BSON 會儲存欄位名稱，但在現代硬體與壓縮技術下，清晰度遠比節省微小的空間重要   
A. 錯誤原因：過度縮寫（如用 a 代表 address）會嚴重降低代碼可讀性，增加維護成本，在現代開發中不建議為省微小空間而犧牲可讀性       
B. 錯誤原因：在同一個文件物件中，欄位的 Key 必須是唯一的，不允許重複定義相同的欄位名稱，且這樣做完全無法優化索引   
C. 正確原因：使用具描述性且唯一的欄位名稱，能讓開發人員快速理解資料結構，是確保專案長期好維護的標準最佳實踐（Best Practice）   
D. 錯誤原因：使用通用、含糊的名稱（如 data1、value）無法明確傳達該欄位儲存的具體業務含意，容易導致混   











[題目]   
What is a key characteristic of MongoDB's document model that allows for handling polymorphic data, or data of different shapes and types? (Select one.)   
   
[選項]   
A. Maximum document size of 16 MB   
B. Similarity to JSON objects     
C. Flexible schema   
D. Single data type storage   
   
[正確答案]：   
C
   
[核心考點]：   
   
[詳細解析]：   
本題考查 MongoDB 處理多態數據（Polymorphic Data）的核心優勢。MongoDB 具備彈性綱要（Flexible Schema）的特性，不強制同個集合內的所有文件必須擁有相同的結構。這允許開發者在同一個集合中，自由儲存結構相異、欄位不同的文件，完美應對多變的業務需求。   
   
A. 錯誤原因：16 MB 是單一 BSON 文件的實體容量限制（用以防止濫用並優化記憶體使用），但該限制本身並非提供結構多態性的核心機制。   
B. 錯誤原因：雖然 MongoDB 採用類似 JSON 的 BSON 格式來直觀地儲存與查詢資料，但「類似 JSON」這項特質本身並不是動態多態結構的直接定義詞。   
C. 正確原因：彈性綱要（無綱要限制 / Schema-less）是 NoSQL 文件資料庫最顯著的特徵，也是實現不同資料形狀與多態設計的基石。   
D. 錯誤原因：MongoDB 支援豐富的資料型別（多元儲存），選項敘述的單一型別儲存（Single data type storage）與其支援多種類型（如字串、數字、陣列、物件）的事實完全相反。   








[題目]

Which of the following value types are supported by MongoDB BSON? (Select one.) 以下哪一組是 MongoDB BSON 支援的資料型別？

[選項]

A. String, Integer, Boolean, Date
B. String, Integer, Boolean, Date, Binary, ObjectId, Array
C. String, Integer, XML, Date
D. String, Integer, Boolean, Set

[正確答案]：
B

[核心考點]：

[詳細解析]：
本題考查 MongoDB BSON 支援的原生型別。BSON（二進位 JSON）在相容 JSON 基本型別的基礎上，擴充了 ObjectId、Binary、陣列等多種特殊二進位與結構型別，提供更完整的型別支援。  
A. 錯誤原因：雖然此選項列出的型別皆被支援，但相較於選項 B，它漏掉了 Array、ObjectId 和 Binary 等重要且常用的原生 BSON 型別。  
B. 正確原因：完整且精確地包含了 BSON 所支援的多種原生型別（字串、整數、布林、日期、二進位、物件標識符與陣列）。  
C. 錯誤原因：XML 並非 BSON 的原生資料型別。若要在 MongoDB 中儲存 XML，必須將其轉換為字串，或是先解析並轉換為 BSON 內嵌文件。  
D. 錯誤原因：Set（集合，且不重複元素）並非 BSON 原生支援的資料型別。在 MongoDB 中通常是以 Array 搭配邏輯控制（如 $addToSet）來實作。  






[題目]  
Which BSON type is automatically generated by MongoDB as the default value for the _id field if not provided? (Select one.) 若未指定 _id，MongoDB 預設會自動產生哪一種 BSON 型別？  

[選項]  
A. String  
B. UUID  
C. ObjectId  
D. Integer  

[正確答案]：  
C

[核心考點]：  
  
[詳細解析]：
本題考查 MongoDB 中 _id 欄位的預設型別。在向集合插入新文件時，若應用程式或驅動程式未手動指定唯一的主鍵 _id 欄位，MongoDB 系統預設會自動產生一個 12 位元組的 ObjectId 作為其值。  
A. 錯誤原因：String（字串）可以用來作為手動指定的 _id 值，但它並非 MongoDB 未指定 _id 時所採用的自動生成預設型別。  
B. 錯誤原因：UUID 是另一種常見的通用唯一識別碼，雖然 MongoDB 也支援儲存 UUID，但系統預設自動生成的 _id 是 ObjectId 而非 UUID。  
C. 正確原因：ObjectId 是 MongoDB 官方為分散式系統設計的 12 位元組（96-bit）唯一識別碼。當寫入資料未主動提供 _id 時，系統預設會自動生成此型別填入。  
D. 錯誤原因：Integer（整數）通常是傳統關聯式資料庫（RDBMS）中常用的自增主鍵型態。在 MongoDB 的分散式場景下，預設的主鍵類型為 ObjectId。  






[題目]

**You have the following three documents. Which of them can coexist in the same MongoDB collection? (Select one.)** *下列三個文件，哪些可以同時存在於同一個 MongoDB collection 中？*

```json
// Document A
{ "_id": 1, "name": "Alice", "age": 30 }

// Document B
{ "_id": 2, "name": "Bob", "email": "bob@test.com" }

// Document C
{ "_id": 3, "name": "Charlie", "age": "thirty" }
```

[選項]

A. nly Document A and B
B. Only Document A and C
C. Only Document B and C
D. Document A, B, and C

[正確答案]：  
D

[核心考點]：  

[詳細解析]：
本題考查 MongoDB 的無綱要（Schema-less）與彈性文件模型。在同一個集合（Collection）中，文件不需要具備相同的欄位與資料型別，因此這三個結構、型別相異的文件可以完美共存。  
A. 錯誤原因：MongoDB 具備動態綱要特性，同一個集合不限於只有 A 與 B 共存，欄位結構不同的 C 同樣可以與它們放在一起。  
B. 錯誤原因：MongoDB 允許同一個欄位（如 age）在不同文件中使用不同的資料型別（如整數與字串），因此不限於只有 A 與 C 共存。  
C. 錯誤原因：除了 B 與 C 外，欄位完全不同的 A 同樣能共存在同一個集合中。  
D. 正確原因：MongoDB 的文件模型不強制要求固定綱要（Schema），同集合內的文件可以有不同欄位與資料型別，故 A、B、C 三者可完美共存。  







[題目]
Which statement best describes MongoDB’s document model? (Select one.)  
下列哪一項最能正確描述 MongoDB 的文件模型？  

[選項]  
  
A. All documents in a collection must have identical fields  
B. Documents are stored in tables with fixed schemas  
C. Documents are stored as flexible JSON-like structures  
D. Documents must follow a predefined relational schema

C  
  
[核心考點]：  

[詳細解析]：  
本題考查 MongoDB 的核心資料儲存模型。MongoDB 是基於文件（Document）模型的 NoSQL 資料庫，其資料是以類似 JSON 的 BSON 二進位格式儲存，具備極高的欄位彈性與可擴展性。  
    
A. 錯誤原因：這是傳統關係型資料庫的特徵。在 MongoDB 中，同一個集合中的不同文件不需要擁有完全相同的欄位。  
B. 錯誤原因：MongoDB 將資料儲存在「集合（Collections）」中，並以文件形式呈現，而非具有固定綱要的「資料表（Tables）」。  
C. 正確原因：MongoDB 採用 BSON（類 JSON 的二進位格式）來儲存數據，允許欄位動態增減，結構極其靈活。

[題目]  
Which of the following statements about the _id field in MongoDB is true? (Select all that apply.)  

[選項]  
A. It provides a unique identifier for each document.  
B. It must be defined explicitly by the user when inserting a document.  
C. If omitted during document insertion, MongoDB generates an ObjectId automatically.  
D. It is required for every document in MongoDB.  
  
[正確答案]：A, C, D  

[核心考點]：

[詳細解析]：
本題考查 MongoDB 中 _id 欄位的主要特性與生成機制。在 MongoDB 中，每個文件都必須擁有唯一的 _id 作為識別碼。此欄位不需由使用者明確定義，若在寫入時省略，系統或驅動程式會自動生成一個唯一的 ObjectId 填入。

A. 正確原因：_id 欄位是集合中每個文件的唯一識別碼，確保資料的唯一性並用於高效檢索。
B. 錯誤原因：_id 不需要明確定義，若省略，MongoDB 會自動生成預設值，因此該敘述錯誤。
C. 正確原因：若在插入文件時未提供 _id，MongoDB 預設會自動為該欄位生成一個唯一的 ObjectId。
D. 正確原因：MongoDB 中的每個文件都必須包含 _id 欄位，它是集合中不可或缺的主鍵。






[題目]  
Which of the following data types is used for high-precision calculations and financial data in BSON? (Select one.)  

[選項]  
A. Boolean  
B. Decimal128  
C. Int32  
D. Null  

[正確答案]：
B

[核心考點]：

[詳細解析]：
本題考查 BSON 資料型別選用。在金融或高精度運算場景中，為避免浮點數精度的二進位轉換誤差，BSON 提供了支援 34 位有效數字的 Decimal128 型別，是高精度計算與貨幣資料的標準選擇。

A. 錯誤原因：Boolean（布林值）僅用於表示 true 或 false 的二元狀態，無法進行任何數學或高精度的數字計算。

B. 正確原因：Decimal128 是 BSON 專門為金融與高精度計算設計的 128 位元十進位浮點數，能完全避免浮點數精度丟失的問題。

C. 錯誤原因：Int32 是 32 位元有號整數，只能儲存整數，無法處理金融資料中常見的精確小數與分數。

D. 錯誤原因：Null 用於表示空值或不存在的值，本身不具備任何數值意義，更無法用於數學計算。

# Lesson 4: Data Relationships


[題目]

In our movies collection, which is an example of a one-to-many relationship? (Select one.)  

[選項]  

A. Movies and the theaters that are showing them.  
B. A movie and the studio that released it.  
C. A movie and its cast members.  
D. Theaters and the movies they are showing.  

[正確答案]：  
C

[核心考點]：  

[詳細解析]：  
本題考查資料庫關係建模。在電影集合中，一部電影（One）會擁有多個演員（Many）參與演出，在單一電影文件中通常以陣列形式內嵌，屬於典型的一對多（One-to-Many）關係。  

A. 錯誤原因：電影與戲院為雙向多重關聯（一部片在多間戲院放，一間戲院播多部片），屬於多對多（Many-to-Many）關係。  

B. 錯誤原因：一部電影通常只對應到一個發行片商，這屬於一對一（One-to-One）關係。  

C. 正確原因：單一部電影與其多位演員的關係是典型的一對多（One-to-Many）關聯。  

D. 錯誤原因：戲院與播映電影的關係同 A 選項，屬於多對多（Many-to-Many）關係。  


# Lesson 5: Embedding and Referencing


[題目]   
What is the typical approach for modeling a one-to-one relationship in MongoDB, where one entity is related to exactly one other entity? (Select one.)   
   
[選項]   
A. Embedding   
B. Referencing   
C. Flattening   
D. Normalizing   

[正確答案]：   
A   
   
[核心考點]：   

[詳細解析]：   
本題考查 MongoDB 中一對一（One-to-One）關係的標準建模方式。MongoDB 的核心設計哲學是「經常一起被存取的資料，應該儲存在在一起」。因此，對於一對一的關係，最典型且首選的作法是內嵌（Embedding），將相關聯的實體作為子文件直接嵌入主文件中，以達成單次 I/O 讀取。   
   
A. 正確原因：內嵌能將資料保持在同一個文件內，簡化查詢邏輯，並確保在單次資料庫讀取中就能完整獲取兩個關聯實體的資訊，是處理一對一關係的標準作法。   

B. 錯誤原因：引用（Referencing）會將資料拆分到不同的集合中，這在查詢時會需要額外的聯查（如 $lookup）或多次請求，增加了架構複雜度與效能開銷。   

C. 錯誤原因：扁平化（Flattening）是指將巢狀結構展開為單層欄位的程序，與探討如何建立兩個獨立實體間的一對一關係模型無關。
   
D. 錯誤原因：正規化（Normalizing）是關聯式資料庫（RDBMS）為了減少資料冗餘的傳統作法，但這與 MongoDB「為了讀取效能而適度去正規化」的設計原則相違背。   



[題目]
You are designing a MongoDB schema for a movie database. There are approximately 40 movie genres represented in the database. A movie will typically belong to a few genres, and each genre could include thousands of movies.   
   
If you want to include the genre information for each movie, while also being able to query for all movies within a genre, how should you model the relationship between movies and genres?   
   
[選項]   
A. Embed the genres directly within the movie document.   
B. Reference the genres in the movie document using an array of genre IDs.   
C. Embed the movie documents within each genre document.   
D. Reference the movies in each genre document using an array of movie IDs.   
   
[正確答案]：   
A   

[核心考點]：

[詳細解析]：
本題考查多對多關係（Many-to-Many）在 MongoDB 中的文件建模決策。評估標準在於基數（Cardinality）與文件大小限制（16MB）。一電影片僅有少數類型（如 ['Action', 'Sci-Fi']），將類型內嵌於電影文件中體積極小，且能透過建立多鍵索引（Multikey Index）高效查詢特定類型下的所有電影。

A. 正確原因：每部電影對應的類型數量極少（低基數），直接內嵌字串或小物件完全不會導致文件膨脹，且能實現單次 I/O 讀取完整電影資訊，是最符合 MongoDB 效益的作法。

B. 錯誤原因：雖然使用 ID 陣列引用可行，但在類型總數極少（僅 40 種）且每部片只佔幾個的場景下，使用引用會被迫進行額外的關聯查詢，多此一舉且降低讀取效能。

C. 錯誤原因：一個類型可能包含數萬部電影（高基數）。若將大量電影內嵌在類型文件中，陣列會急遽膨脹，極易觸犯 16MB 的文件上限並嚴重衝擊記憶體與寫入效能。

D. 錯誤原因：在類型文件中存放成千上萬個電影 ID 陣列（稱為反向引用），同樣會面臨陣列過大、更新頻繁時容易導致文件搬移與效能低落的「大陣列反模式（Large Array Anti-pattern）」。




[題目]	
In MongoDB document modeling, when representing a one-to-one relationship between a 'user' and their 'address', a developer must choose between embedding the address subdocument inside the user document versus referencing it via a separate collection. Given that the address data is always queried together with the user, is never shared with other documents, and the combined document size stays well below MongoDB's 16MB limit - but the address subdocument itself is very rarely updated independently - which modeling approach is most appropriate and why? (difficulty: medium)
	
[選項]	
A. Embed the address subdocument directly inside the user document, because the data is always accessed together, is not shared across documents, and fits within size constraints - making embedding the optimal choice for this access pattern.	
B. Embed the address only if the address fields are frequently updated independently, because frequent independent updates are the primary reason to choose embedding in MongoDB.	
C. Use a reference (store the address in a separate 'addresses' collection and keep an address_id field in the user document), because referencing always provides better write performance regardless of access patterns.	
D. Use a reference, because one-to-one relationships in MongoDB must always be modeled as separate collections to maintain data normalization similar to relational databases.	

[正確答案]：
A

[核心考點]：

[詳細解析]：
本題考查 MongoDB 的資料建模設計。在一對一關係中，若子數據（如地址）總是與主數據（如用戶）一同讀取、不被其他文檔共享且大小未達 16MB 限制，內嵌（Embedding）是最佳實踐，能有效消除 Join 查詢開銷。

A. 正確原因：內嵌子文件能將常共同存取的欄位封裝於單一文件中，使資料庫僅需一次磁碟 I/O 即可讀取全部資訊，完美契合「一同存取」的應用場景。

B. 錯誤原因：頻繁且獨立的更新通常更適合「引用（Referencing）」，因為這樣做能避免主文件頻繁重寫與鎖競爭；若為此原因選擇內嵌，邏輯完全倒置。

### Question 7
*Which query returns all documents where the age field exists, regardless of its value? (Select one.)
下列哪個查詢可回傳所有包含 age 欄位的文件，不論其值為何？*

[選項]  
A. { age: { $exists: true } }
B. { age: { $exist: true } }
C. { age: { $ne: null } }
D. { age: true }
	
[正確答案]：
A	
	
[核心考點]： 	
	
[詳細解析]：
說明：本題考查判斷欄位是否存在的查詢操作符。`$exists` 是唯一正確用於檢查欄位存在性的運算子，能匹配所有包含該欄位的文件（不論值為 null 或其他類型）。

A. 正確原因：`$exists: true` 是 MongoDB 判斷欄位是否存在的正確語法，不論欄位值為何皆能正確比對並回傳。

B. 錯誤原因：MongoDB 中並無 `$exist` 此操作符，這純屬拼寫錯誤的無效語法。

C. 錯誤原因：此條件會同時排除欄位值為 `null` 的文件，因此無法滿足「不論其值為何（即使是 null 也要回傳）」的條件。

D. 錯誤原因：這代表等值查詢（Equality match），只會尋找 `age` 欄位值恰好為布林值 `true` 的文件，而非判斷欄位是否存在。




[題目]  
### Question 8
*Which index type supports equality matches and range queries efficiently?*

[選項]  
A. Text index
B. Hashed index
C. B-tree index
D. TTL index
	
[正確答案]：
C	
	
[核心考點]： 	
	
[詳細解析]：
說明：本題考查 MongoDB 的底層索引結構特徵。標準單欄位與複合索引底層均採用 B-tree（確切為 B+ tree）結構，因其節點資料具備有序性，故能同時高效支援等值與範圍查詢。

A. 錯誤原因：Text（文本）索引專門用於字串的全文檢索，不支援常規的等值比對，更不支援範圍查詢。

B. 錯誤原因：Hashed（雜湊）索引在計算雜湊值後資料會失去原有的順序性，因此它僅能用於等值比對，完全不支援範圍查詢。

C. 正確原因：MongoDB 的預設索引結構為 B-tree，節點內的鍵值是有序排列的，因此能同時保證等值搜尋與範圍掃描（如大於、小於）的高效執行。

D. 錯誤原因：TTL 索引是一種特殊索引，主要作用是根據設定時間自動過期並刪除文件，其底層雖也是 B-tree，但非專門解決常規等值與範圍查詢的索引類型。




[題目]      
Which statement about indexes is TRUE?      

[選項]      

A. Indexes improve query performance with no trade-offs
B. Indexes reduce disk I/O
C. Indexes slow down all queries
D. Indexes are auto-created by MongoDB

[正確答案]：
B

[核心考點]：

[詳細解析]：
說明：本題考查 MongoDB 索引的基本運作原理與效能權衡（Trade-offs）。索引的主要目的在於避免資料庫進行代價高昂的全表掃描（COLLSCAN），改用高效的索引掃描（IXSCAN），進而大幅降低磁碟讀取次數（Disk I/O）。

A. 錯誤原因：索引是有代價的（Trade-offs）。建立索引會佔用額外的記憶體與硬碟空間，並且在每次執行寫入、更新或刪除操作（Write operations）時，資料庫都必須同步維護索引，因而會輕微降低寫入效能。

B. 正確原因：在沒有索引時，MongoDB 必須從磁碟將整張集合的所有文件載入記憶體比對。有了索引後，MongoDB 只需要載入少量的索引樹節點並精準定位目標文件，大幅降低了不必要的磁碟讀寫（Disk I/O）。

C. 錯誤原因：恰恰相反，設計良好的索引能讓大部分符合條件的讀取查詢（Read queries）效能提升數十甚至數百倍，絕對不會慢下來。

D. 錯誤原因：除了預設的 _id 欄位索引是由系統自動建立外，其餘所有客製化查詢所需的索引（如 name、age 等），都必須由開發人員根據具體的查詢模式手動設計與建立。




[題目]  
In MongoDB one-to-many relationship modeling, a product catalog system stores products, and each product can have thousands of customer reviews. A developer is choosing between embedding reviews inside the product document versus referencing them in a separate collection. Which of the following statements best describes a critical limitation of the embedding approach in this specific scenario? (difficulty: medium)
在 MongoDB 的一對多關係建模中，一個產品目錄系統存儲產品，且每個產品都可能擁有數千條客戶評價。開發人員正在考慮要將評價內嵌在產品文件內，還是將它們引用存儲在另一個獨立集合中。在這種特定場景下，下列哪一個敘述最準確地描述了內嵌方法的一個關鍵限制？（難度：中等）

[選項]  
A. Embedded documents cannot store nested arrays, so review data with sub-fields like 'rating' and 'comment' cannot be represented inside a product document.
B. Embedding reviews inside a product document prevents the use of indexes on review fields, making it impossible to query reviews efficiently.
C. Embedding thousands of reviews inside a product document risks exceeding MongoDB's 16 MB per-document size limit, degrading performance and preventing new reviews from being added.
D. Embedding reviews inside a product document requires all reviews to share the same schema, which conflicts with MongoDB's schema-less nature and causes write errors.
	
[正確答案]：
C	
	
[核心考點]： 	
	
[詳細解析]：
說明：本題考查一對多關係中的「無限制增長」問題。若將數千條評價內嵌於單一文件中，極易突破 16MB 的硬性限制，且頻繁的陣列追加寫入會導致嚴重的磁碟重新分配與效能衰退，故此場景應採用引用模式。

A. 錯誤原因：MongoDB 完全支援嵌套陣列與嵌套文件（Embedded Documents），評價中的 rating 與 comment 等子欄位可以被完美呈現，此非內嵌的限制。

B. 錯誤原因：MongoDB 支援在內嵌文件的欄位上建立多鍵索引（Multikey Index），例如建立 `reviews.rating` 索引，依然能進行高效查詢，並非無法使用索引。

C. 正確原因：MongoDB 的單一文件大小上限為 16MB。當評價多達數千條時，不僅文件極易超過此限制導致無法寫入，頻繁更新大文件也會造成嚴重的記憶體與磁碟 I/O 開銷。

D. 錯誤原因：MongoDB 是無綱要（Schema-less）的，內嵌文件不需要共享相同的綱要，不同結構的評價並不會導致寫入錯誤。
